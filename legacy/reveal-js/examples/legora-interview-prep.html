Platform depth

Permission leakage: authorize before retrieval.

Make this answer crisp. It is directly relevant to legal work.

Core answer

  • Normalize identity, tenant, matter, role, and document ACLs.
  • Enforce filters in both keyword and vector retrieval.
  • Use synthetic cross-matter fixtures for testing.
  • Log every source access for auditability.

Integration answer

  • Center the domain model on the matter and document bundle.
  • Keep adapters at the edges: Word, Outlook, DMS, matter systems.
  • Use idempotent writes, retries, and explicit side-effect boundaries.
  • Preserve a testable, auditable core workflow engine.

The model should never see text the user is not authorized to see.

System design prompt

Agentic legal review workflow.

Start permission-aware. Do not start with an autonomous black box.

Prompt

A lawyer receives a contract redline or client request. Design a system that reviews changes against matter context, firm playbooks, precedent, and authoritative sources; proposes a response; and routes it for lawyer approval before delivery.

Walkthrough

  1. Clarify inputs, outputs, risk boundary, and approval gates.
  2. Normalize documents into a canonical matter bundle.
  3. Authorize before retrieval.
  4. Plan explicit subtasks and use typed tools.
  5. Validate citations, schemas, rules, and confidence.
  6. Capture traces and corrections as eval cases.
The applet's verbal walkthrough: normalize matter bundle, authorize before retrieval, retrieve source-grounded context, plan explicit subtasks, use typed tools, validate outputs, route sensitive delivery through lawyer approval, and turn traces into audit records and eval cases.

Architecture map

Legal review agent: visible control boundaries.

Use this order under pressure.

Node notes from applet: intake starts with matter request and bundle; normalize files, metadata, versions, and citations; authorization is a precondition to retrieval and generation; context is source-grounded and permission-allowed; planner must stay inspectable; tools are narrow and typed; validate before lawyer view; approval is the safety boundary; delivery returns to existing workflow; traces feed auditability, debugging, and regression evals.

Risk and evaluation

Use senior language: threat model, evals, release gates.

This is where the answer becomes production-grade.

Failure modes

  • Retrieval leakage across matters or tenants.
  • Citations that do not support generated claims.
  • Silent truncation or missed document versions.
  • Partial failure after external integration writes.
  • Overconfident output when sources conflict.

Threat model

  • Prompt injection from documents or emails.
  • Sensitive information disclosure.
  • Excessive agency without approval.
  • Unsupported claim appears client-ready.
  • Unbounded cost or latency from large bundles.
Tradeoffs from applet: async processing helps large document reliability but needs status UX and resumable jobs; narrow tools reduce blast radius and improve evals but require more orchestration; conservative approval gates slow full automation but fit legal trust-building.
